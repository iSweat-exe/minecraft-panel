# Rust Status & Doc Access — AI Skill

## Purpose of this file

AI models are trained on data frozen at a given cutoff date. As a result, they often
assume a Rust version, edition, or feature availability that matches their training
cutoff (e.g. "Rust 1.76", edition 2021) even when the project is actually on a much
more recent version.

This file has 3 goals:
1. Force verification of the actual Rust version instead of guessing it.
2. Provide a method for staying current on new features (stabilized features, lints).
3. Give access to local Rust documentation (std + project dependencies), to avoid
   hallucinating API signatures.

**This file is not static.** It's meant to be edited on every version bump
(1.98, 1.99, etc.). Section 5 = log to keep up to date.

## How to wire it into different tools

This file is tool-agnostic. A few ways to reference it:
- **Claude Code**: put it at the repo root or in `~/.claude/skills/rust-status/SKILL.md`,
  and reference it from your `CLAUDE.md` (`@rust-status-skill.md`).
- **Cursor**: reference it in `.cursor/rules/` or `.cursorrules`.
- **Other agents (Antigravity, etc.)**: reference it in whichever instructions file
  it reads first (often `AGENTS.md` or equivalent).

## 1. Current Rust version — NEVER assume, always verify

### Dynamic check (top priority whenever a terminal is available)

Run this **before** making any claim about available syntax/edition/API:

```bash
rustc --version --verbose
cargo --version
rustup show active-toolchain
grep '^edition' Cargo.toml
```

`rustc --version --verbose` gives the exact version + commit hash + build date.
`grep edition Cargo.toml` gives the edition actually used by THIS project (it can
differ from the latest edition available).

### Manual fallback (if no terminal access / purely conversational context)

If the AI has no access to a terminal, it must read the field below and
**explicitly flag** that it's relying on potentially stale information:

```
Last known version : 1.97.0 (stable)
Project edition     : 2024
Last update of this field : 2026-07-15
```

Rule: edition (2015/2018/2021/2024) and compiler version (1.XX) are two
independent things. A feature can be stabilized in a given rustc version
without depending on the edition, and conversely some syntax/default changes
only apply starting from a given edition. Don't conflate the two.

## 2. Staying current on new features

When the AI isn't sure whether a feature/API/lint exists in stable, it should
**search** rather than assume (in either direction — don't default to saying
"it doesn't exist," and don't invent an API that doesn't exist yet either).

Sources to check, in order of reliability:

1. **Official RELEASES.md**: `rust-lang/rust` on GitHub, `RELEASES.md` file —
   exhaustive and exact list of what was stabilized in each version.
2. **Official blog**: blog.rust-lang.org/releases — detailed announcements per version.
3. **Edition Guide**: doc.rust-lang.org/edition-guide — specifically for changes
   tied to an edition (2021 → 2024 for instance).
4. **Recent Clippy lints**: `cargo clippy -- -W clippy::all` + Clippy changelog
   (often tied to new rustc versions).
5. Targeted web search like "Rust 1.97 release notes" if the AI has web access.

## 3. Local Rust documentation access

### Initial setup (once per machine)

```bash
# std/core/alloc docs — normally included by default with rustup's "default"
# profile, but worth checking/installing explicitly if needed:
rustup component add rust-docs

# The std source itself — useful for reading the actual implementation
# instead of guessing a function's behavior:
rustup component add rust-src
```

### std/core docs (browser)

`rustup doc` opens the index in the default browser. Useful flags:

```bash
rustup doc --std             # std docs
rustup doc --book            # The Rust Book
rustup doc --reference       # The Rust Reference
rustup doc --cargo           # Cargo docs
rustup doc --rustc           # compiler docs
rustup doc --edition-guide   # edition guide
rustup doc --path            # just print the local path, don't open
```

### Local access without a browser (for an agent reading files)

The docs generated by `rustup doc --path` are static HTML. An agent can
browse/grep the folder directly:

```bash
# Find the actual folder (depends on the active toolchain):
ls ~/.rustup/toolchains/*/share/doc/rust/html/std/
```

For debugging code that doesn't compile, faster and 100% offline:

```bash
rustc --explain E0499   # plain-text explanation of an error code
```

### Project dependency docs (the most important point)

Online docs (docs.rs) generally show the **latest published version** of a
crate — not necessarily the one actually pinned in `Cargo.lock`. For crates
like `windows-rs`, `retour`, `directx_hooks_rs`, `cef-rs`, this can quickly get
the AI out of sync (it reads an API that isn't the one used in the project).

Best practice: generate the docs **locally**, with the lockfile's exact versions:

```bash
cargo doc --no-deps --open        # docs for the current crate only
cargo doc --open                  # docs for the crate + all its dependencies
cargo doc --document-private-items --open   # includes private items (useful on your own code)
```

Output goes to `target/doc/<crate_name>/index.html`. An agent without a browser
can grep the generated HTML files in `target/doc/` to find an exact signature
instead of relying on its training memory.

## 4. Expected AI behavior (summary)

- Verify the actual version/edition (section 1) before making any claim about
  what is or isn't "possible" in Rust.
- If only the manual field in this file was used (no terminal available),
  explicitly say so to the user.
- Don't conflate compiler version and edition — they're two independent axes.
- Before saying a feature doesn't exist in stable: verify via RELEASES.md /
  official docs / web search, don't decide from memory.
- For any question about a project dependency (API behavior, signature, version),
  prefer locally generated `cargo doc` (pinned versions) over a generic web search.
